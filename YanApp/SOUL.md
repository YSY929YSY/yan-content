# 言审美手册

## 言的气质

言是精简克制，不是温情叙事。目标用户是审美挑剔、有自己主见的文艺向旅行者——浪漫感靠精准和留白传递，不靠铺陈。

**测试句**：读完某段文案，问自己："这句话是在说一件事，还是在表演一种语气？"后者直接删。

禁用句式：
- 「X时同样用」「日本人会说X表示Y」「X的核心功能是」
- 「让XX有了温度」「在旅途中，总有这样的时刻……」
- 任何把已经展示的内容再用文字总结一遍的句子

参考好的语气：「注文是行动信号，不只是"点餐"。你说出这个词，等于告诉店员：我选好了，可以来了。」


## 气质不靠文案表演

言的气质来自内容判断、分类方式、信息层级、留白和克制的视觉秩序，不靠把普通功能文案改成更有感觉的句子。

功能按钮优先使用短、明确、低解释成本的词：
- 认识 / 不认识
- 返回
- 下一词
- 搜索
- 全部
- 学习中
- 已掌握

不要把清楚的功能文案改成诗化、心理化、安慰式表达：
- 还没抓住
- 已经能认出
- 再靠近一点
- 让这个词留下来

这些句子会占空间、降低效率，并让言显得在表演气质。

判断标准：如果一个文案只是把简单功能说得更有情绪，而没有增加理解、判断或行动价值，就删掉，换回短词。

言的高级感应该藏在：
- 词为什么这样分类
- 哪些词值得进入言库
- 例句是否真实
- 语域是否准确
- 视觉是否安静
- 信息层级是否清楚

不是藏在按钮文案里。


## UI 判断以真实渲染为准

改 UI 时必须按照 rendered state 判断，不按照 computed state 自我安慰。代码里的样式、设计意图、computed layout 都不能替代真实截图或模拟器画面。

每次改界面后至少检查：
- 真机或模拟器截图里是否真的好看
- 文字是否拥挤、溢出、遮挡或换行难看
- 卡片、按钮、列表密度是否符合言的克制气质
- 主要信息是否一眼能读，次要信息是否退后
- 底部导航、顶部 tab、输入框、按钮是否互相抢层级

如果截图里丑，就按截图修；不要用“代码上应该没问题”“computed style 是对的”来通过验收。

## 词卡九标准（顺序即优先级）

| # | 标准 | 自问 |
|---|------|------|
| 1 | **真实** | 旅行者真实会遇到，不是课本造的？ |
| 2 | **内容** | 核心义、陷阱、例句都有吗？ |
| 3 | **意象** | 这个词有没有核心画面？一句话说出它的"感觉"？ |
| 4 | **深度** | 有比字面义更深一层的东西吗？ |
| 5 | **不赘** | 同一信息有没有说两遍？ |
| 6 | **简洁** | 每句能更短吗？能删的字有没有留着？ |
| 7 | **节奏** | 换行在对的地方吗？读出来喘得过气吗？ |
| 8 | **实用** | 真实场景里能用上吗？ |
| 9 | **人类** | 读起来像老师说话，还是 AI 总结？ |

## 意象的正确用法

意象不是临场编一个画面感场景，是去查词典/词源，找到**同一个词不同语义之间共享的那个核心画面**。

判断要不要加意象的顺序：
1. 先问：这个词翻成中文/对应的汉字，是否已经自然地统一了它的各个义项？如果是，汉字本身就是锚，不要再叠加意象（例：「通る」→ 中文"通过"已经统一所有用法，不需要再画图）
2. 只有当翻译把统一性打散成几个看起来无关的中文词时，才需要去挖那个被遮住的图（例：「toll」→ "过路费/死亡人数/敲钟"，需要找回共通核心）
3. 找不到真实存在的核心意象，宁可留白，不要硬凑

**没有就不要加**。格式完整不是目标，言简意赅才是。

## 内容设计原则

**例句规则**：默认显示3条，第4条起折叠。👂 = 对方说的，🗣 = 你说的。

**grammar depth 选型**：
- 词根/形态变化 → `type:'morph'`（pill chain）
- 词义对比（vs. X）→ `type:'compare'`（两列卡）
- 其他 → 纯文本 block
展示了对比结构，就不再用文字解释"因为A所以B"。

**分层原则**：第一层核心义（简洁），第二层进阶义用「顺带一提」折叠。

**多义词介绍顺序**：先给义项地图（这个词能落在哪些语境上），例句是用来佐证地图的，不能先于地图出现。


## 例句设计规则

例句不是词条的附属字段，而是词进入真实语言的入口。基础词书里的核心例句要先做到：目标词突出、短、真实、中文自然、适合朗读跟读。

N5-N4 阶段，例句罗马音是有价值的。它让例句、TTS、跟读和发音学习对齐，不是低级替代。可以优先为核心例句补 `exampleRoma`，或后续拆成 `exampleParts` 时为每个片段提供读音辅助。

例句呈现的优先级：
1. 日文原句
2. 罗马音 / 读音辅助
3. 中文自然义
4. 必要时再做分词拆解

不要为了显得专业而先堆语法说明、按钮或工具入口。用户先要读懂、听懂、跟得上。

## 例句与词网的后续方向

以下方向有价值，但不要在基础词书 UI 没稳定前一次性加入：
- `exampleParts`：把例句拆成词块，显示片段义和读音辅助。不要靠 AI 手切，必须由 SudachiPy / pykakasi 等工具生成候选，再用审计报告和人工判断修例外。
- `examplePattern`：标注可迁移句型，如 `〇〇はどこですか`、`〇〇をください`
- `exampleDomain`：标注 travel / daily / business / anime / literature / trade 等语境
- `exampleRegister`：标注 neutral / polite / casual / formal / written / literary 等语域
- `chunks`：一个词的常用词组网络，不只保留单个 coreChunk
- 词网 tab：同场景、易混、同类、言库入口

V2 例句增强路线：
1. 先显示整句 `exampleRoma`：日文原句 → 罗马音 → 中文自然义。
2. 再用工具生成 `exampleParts`，做下划线 + 片段读音；App 只渲染内容字段，不在前端现场分析日语。
3. 最后形成言自己的特色：给片段增加轻量 `role`，例如 time / object / action / place / request / judgment，让用户看见一句话的行动结构，而不是照搬语法课式拆解。

行动结构是一种轻语法，但不要用传统语法术语吓用户。传统语法会说“时间状语 / 宾格助词 / 他动词ます形”；言优先说“时间 / 对象 / 动作”。目标是让用户先看懂一句话如何完成一个行动，再自然迁移到新句子。

例：`朝、コーヒーを飲みます。` 可以拆成：
- `朝`：time / 时间 / asa
- `コーヒー`：object / 对象 / koohii
- `飲みます`：action / 动作 / nomimasu

这样同时完成三件事：发音辅助、句子理解、语法预埋。不要把它做成语法课，也不要堆术语按钮。

下划线例句的目标不是炫技，而是让用户把声音、词块和行动结构对齐。宁可先少量高质量，也不要全量粗糙铺开。

这些是言未来的学习深度，不是现在改 UI 时可以随手堆上的装饰。只有当内容字段稳定、基础词书界面安静、例句质量过关后再进入界面。

## 视觉身份规则

**暗色风格（深色背景/毛笔/ink-lava配色）** = 言自己原创的深度内容（深度词卡、三层连接模块等）

**白底轻卡（ls.card / wbs.row）** = 通用功能入口（词书选择、词书封面、练习入口等）

判断：这个内容是"言创作的"还是"言导入的/聚合的"？  
导入的走白底轻量风格，不要穿言的暗色外衣。

**骨架模块结构**：展示槽 = JP 变量（学的东西）；选择 chip = 中文标签（导航用）；底部轻字体显示完整句。两者职责不重叠，标题是功能性标签，不做文案（"换个需求"，不是"出门在外用它包办一切"）。

## 中文措辞语感

中文释义不只是字面对应准确，要检查母语者联想：
- "算账"带有火药味（找你算账），"结账"是平和收尾 → 用"结账"
- 写完中文释义问一步：母语者听到这个词，第一反应是什么情绪？

用户已经明确过的具体措辞不要"顺手"改回字面直译版本——那是 regression，不是改进。
